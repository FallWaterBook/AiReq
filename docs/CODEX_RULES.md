# CODEX_RULES.md (AiReq / OpenAI API multi-file JSON output rules)

この文書は、AiReq で OpenAI API モデルに修正依頼する際の最上位運用ルールです。  
本ルールの目的は、複数ファイル修正を安全に扱うために、厳格な JSON 出力契約と変更範囲制御を定義することです。

## 1. 出力契約 (最重要)

- 出力は **JSON オブジェクトのみ**。
- 出力ルートは `{"files": [...]}` 形式のみ。
- Markdown、説明文、見出し、要約、コードブロックを出力してはならない。
- diff 形式を出力してはならない。
- JSON として parse 可能であること。
- trailing comma を含めてはならない。

## 2. JSON安全性強化

- `content` 内の文字列は JSON として安全でなければならない。
- ダブルクォート (`"`) は必ずエスケープすること。
- 改行は `\n` としてエスケープすること。
- バックスラッシュは適切にエスケープすること。
- JSON が壊れる可能性がある場合は `{"files": []}` を返すこと。

## 3. 複数ファイル制限

- `files` 配列に含められるのは最大 3 ファイル。
- 変更対象は TARGET_REPO_DIR 配下のみ許可。
- 危険ディレクトリ（.git, .venv, venv, node_modules, db.sqlite3, media, staticfiles, __pycache__ 等）は禁止。
- 絶対パスは禁止。
- `..` を含むパスは禁止。

## 4. files 要素のスキーマ

- 各要素は次の形式に厳密一致すること。
- `{"path": "...", "content": "..."}`
- `path` は文字列。
- `content` は文字列。
- 追加キーは出力してはならない。

## 5. File Integrity Rule

- 各 `file` 要素は独立して完全であること。
- `content` が空文字列であってはならない。
- `content` が不完全であってはならない。
- 1つでも不正な `file` がある場合は `{"files": []}` を返すこと。

## 6. Atomic Safety Rule

- `files` 配列は「全成功 or 全失敗」とする。
- 一部だけ成功は許可しない。
- 1つでも安全に生成できない場合は全体を `{"files": []}` にする。

## 7. Size Safety Rule

- 各 `content` は明らかに不正な短さの場合は禁止する。
- 例: 1〜2文字だけの `content` は異常とみなし、`{"files": []}` を返すこと。

## 8. content ルール

- `content` は対象ファイルの修正後全文であること。
- 部分コード、抜粋、差分片は禁止。
- 既存コード構造を可能な限り維持すること。
- 余計なリファクタリングは禁止。
- コメント追加・削除・書き換えは明示指示時のみ許可。
- 不要な空白・改行のみの変更は禁止。

## 9. スコープ制約

- 変更対象は TARGET_REPO_DIR 配下のみ。
- 危険ディレクトリ・危険ファイルの変更は禁止。
- 既存 API 仕様、レスポンス形式、例外意味、既存テスト意図を壊さないこと。

## 10. Safe Failure Mode

- 安全に修正できない場合は `{"files": []}` を返すこと。
- 複数ファイル修正が必要で 3 ファイルを超える場合は `{"files": []}` を返すこと。
- 要件が曖昧で安全に確定できない場合は `{"files": []}` を返すこと。
- 説明文は返してはならない。

## 11. 優先順位

1. JSON として正しいこと
2. JSON 安全性と原子性を満たすこと
3. `files` が安全範囲内であること
4. `content` が全文であること
5. 既存仕様を壊さないこと
6. 最小変更であること
